# Prompt Engineering Mastery Review Guidelines

## Purpose & Scope
These guidelines provide a comprehensive framework to review the **Prompt Engineering Mastery** course. They ensure consistency, factual accuracy, pedagogical quality, and originality across the curriculum.

---

## Review Types
The course will be evaluated through three complementary lenses, mirroring the existing report suite:

1. **Structural Review** – assesses overall course organization, learning objectives, content relevance, and alignment with industry standards (as captured in `course_review.md`).
2. **Pedagogical & Technical Critique** – examines teaching methods, dual‑track design, code quality, security considerations, and depth of technical coverage (as captured in `course_critique.md`).
3. **Originality & Plagiarism Assessment** – verifies that all material is original, properly cited, and free from copyright concerns (as captured in `course_plagiarism.md`).

---

## Review Workflow
1. **Pre‑review checklist** – ensure repository builds, CI passes, and documentation is present.
2. **Execute each review type** – use the corresponding checklist and criteria below.
3. **Consolidate findings** – compile a unified feedback report using the provided template.
4. **Score & Approve** – assign an overall rating and decide on publication.

## Evaluation Criteria (5‑point scale)
| Criterion | Description |
|-----------|-------------|
| **Accuracy** | All statements reflect the state of LLM technology as of 2026. |
| **Learning Objectives** | Clear, measurable outcomes for each lesson. |
| **Hands‑on Labs** | Executable code samples, reproducible steps, and real‑world relevance. |
| **Code Quality** | Style, linting, security best practices, and up‑to‑date SDK usage. |
| **Security Considerations** | Threat modeling, safe defaults, prompt injection defenses. |
| **Accessibility** | Captions, alt‑text, readable fonts, color contrast. |
| **Instructional Design** | Logical flow, pacing, engagement mechanisms (dual‑track, concept checks). |
| **Originality** | No un‑attributed duplication; proper citations for all external concepts. |

---

## Scoring Rubric
- **5 – Excellent**: Exceeds expectations; no revisions needed.
- **4 – Good**: Minor optional improvements.
- **3 – Satisfactory**: Acceptable but several actionable suggestions required.
- **2 – Poor**: Significant issues; must be addressed before publishing.
- **1 – Unacceptable**: Fundamental flaws; reject until reworked.

## Checklists
### Structural Review Checklist
```markdown
- [ ] Course outline matches learning objectives.
- [ ] Dual‑track (conceptual vs. technical) clearly delineated.
- [ ] All modules have consistent naming and versioning.
- [ ] Up‑to‑date references to LLM models and APIs.
```

### Pedagogical & Technical Critique Checklist
```markdown
- [ ] Concept checks vary in format (multiple‑choice, reflection, quizzes).
- [ ] Transition language is conversational, not templated.
- [ ] Code samples use the latest SDKs (OpenAI, Anthropic, Gemini) and follow PEP‑8.
- [ ] Error handling and security guards are demonstrated.
- [ ] RAG and prompt caching sections are present and accurate.
```

### Plagiarism & Originality Checklist
```markdown
- [ ] All external ideas are cited with proper academic references.
- [ ] No verbatim copying from public tutorials or documentation.
- [ ] Custom examples (e.g., meeting transcript) are original.
- [ ] Licensing and fair‑use statements are included where needed.
```
